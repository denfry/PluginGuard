# PluginGuard — план продуктизации и монетизации

> Цель: построить устойчивую монетизацию **так, чтобы базовая проверка «безопасен ли
> плагин» осталась бесплатной и открытой для всех**. Деньги берём за масштаб (API для
> маркетплейсов/хостингов), автоматизацию (мониторинг/CI), дорогой compute (динамическая
> песочница) и enterprise-нужды (self-host / white-label / threat-фид).

## Честная рамка (важно)

- Бесплатный тир — это не «урезанная демка», а **ядро ценности и двигатель роста**. Он же
  собирает данные для reputation-базы (чем больше сканов, тем сильнее ров).
- Никаких обещаний «100% safe». Везде формулировка — **оценка риска**, не гарантия. Это и
  юридическая защита, и честность security-продукта (см. рамку в [PLAN.md](PLAN.md)).
- Главный экономический факт проекта: **статика дёшева, песочница дорога, данные дороги в
  поддержке**. По этой линии и проходит paywall.

---

## Что остаётся бесплатным навсегда (граница доступности)

- Загрузка одного файла на сайте → полный **статический** вердикт, score, findings.
- Публичная шарабельная ссылка на отчёт (несёт бренд = бесплатный маркетинг).
- Open-source движка анализа (доверие к security-инструменту; ров — это данные и хостинг
  песочницы, а не код).

Платное начинается там, где появляется **масштаб, автоматизация, дорогой compute или
enterprise-контур**.

## Текущее состояние (baseline для монетизации)

- `ScanController` — чистый синхронный REST (`POST /api/scan`, `GET /api/scan/{id}`), без
  аутентификации и без учёта вызовов.
- `ScanStore` — **in-memory**, `MAX_REPORTS = 500`, отчёты теряются при рестарте. ← блокер №1
  для истории, мониторинга и биллинга по факту.
- `SandboxService.attach(...)` — уже **асинхронный** (статический отчёт отдаётся сразу,
  песочница досчитывается в фоне). Хорошая основа для job-модели.
- `AnalyzerProperties.SupplyChain` — CVE (OSV.dev) + reputation по SHA-256, с кэшем и
  `offline` kill-switch. Готовая основа для **continuous monitoring** и **threat-фида**.
- `Dockerfile` + `docker-compose` + `output: standalone` + `offline` → почти готовый
  **self-host / air-gapped** продукт.

### Блокеры, которые нужно снять до любой монетизации
1. **Нет персистентности** (`ScanStore` in-memory) — без БД нет истории, аккаунтов, метрик.
2. **Нет аккаунтов/ключей/учёта** — нечем тарифицировать и нечего показывать в личном кабинете.
3. **Нет биллинга** — нечем принимать деньги.
4. **Деплой не разнесён** — фронт и бэк нужно развести (см. ниже): `web` → Vercel, `analyzer`
   → контейнерный хост; песочница требует Docker-in-Docker.

## Phase 0 — Фундамент (без него монетизации не существует)

> Чистая инженерия, ещё без денег. Снимает блокеры №1 и №4.

### 0.1 Персистентность отчётов (PostgreSQL)
- Заменить `ScanStore` на репозиторий поверх Postgres (Spring Data JPA + **Flyway**-миграции).
- Таблицы: `scan` (id, file_name, sha256, artifact_type, score, verdict, created_at, owner_id?,
  report JSONB), индекс по `sha256` (дедуп/реюз) и `owner_id`.
- Сохранять полный `ScanResult` как JSONB (быстрый старт) + вынести ключевые поля в колонки
  для выборок/статистики.

### 0.2 Объектное хранилище загруженных артефактов
- Сырые `.jar`/`.zip` → S3-совместимое хранилище (AWS S3 / Cloudflare R2 / MinIO для self-host).
- Нужно для **re-scan** (мониторинг гоняет старый артефакт по новому threat-фиду) и для
  отложенной песочницы.
- **Приватность как фича**: явная политика хранения/удаления (TTL, opt-out). Люди грузят
  проприетарные плагины — это и риск, и аргумент в пользу self-host.

### 0.3 Модель асинхронных задач (jobs)
- Таблица `scan_job` (status: QUEUED/RUNNING/DONE/FAILED, type: STATIC/SANDBOX/MONITOR).
- Формализовать уже существующий async-паттерн `SandboxService` в общий job-раннер
  (старт — Spring `@Async` + DB; позже — Redis/RabbitMQ при росте нагрузки).
- `GET /api/scan/{id}` отражает статус job'а (уже частично так для песочницы).

### 0.4 Разнесённый деплой
- `web` (Next.js) → **Vercel**. `NEXT_PUBLIC_API_URL` (build-time) → домен бэкенда.
- `analyzer` → контейнерный хост (Render / Railway / Fly.io / Cloud Run / VPS). Dockerfile готов.
- ⚠️ Песочница требует **Docker-in-Docker** → managed-PaaS (Cloud Run/Fargate) обычно не дают;
  под Phase 3-функциональность нужен VPS/выделенный раннер с доступом к Docker-сокету.
- CORS: выставить `PLUGINGUARD_CORS_ALLOWEDORIGINS` на домен Vercel.

**Критерий готовности:** отчёты переживают рестарт; артефакты в объектном хранилище с TTL;
фронт на Vercel дёргает бэк на контейнерном хосте; статус сканов виден по job-модели.

## Phase 1 — Аккаунты и API-платформа (открывает B2B — главные деньги)

> У тебя уже чистый REST. До коммерческого API остаётся: личность, ключ, учёт, лимит, счёт.

### 1.1 Аутентификация и организации
- Сущности `User` и `Organization` (членство, роли owner/member). Тарифицируем **организацию**.
- Реализация прагматично: Spring Security на бэке; пользовательский вход можно делегировать
  внешнему провайдеру (Clerk / Auth0 / Supabase Auth) на стороне `web`, бэк проверяет JWT.

### 1.2 API-ключи
- Таблица `api_key` (hash ключа — храним только хэш, prefix для отображения, org_id, scopes,
  created/last_used, revoked). Заголовок `Authorization: Bearer pg_live_...`.
- Spring `OncePerRequestFilter`: резолвит ключ → org → тариф; кладёт в контекст запроса.

### 1.3 Учёт использования и квоты
- Таблица `usage_event` (org_id, endpoint, units, ts) — пишется на каждый платный вызов.
- Rate-limit и квоты по тарифу (Bucket4j или Redis token-bucket). Разные лимиты для
  static / sandbox (sandbox дороже → строже).
- Заголовки `X-RateLimit-*` в ответе.

### 1.4 Биллинг (Stripe)
- Stripe: подписки (Pro) + **metered usage** (API сверх квоты) + webhooks (`checkout.session`,
  `invoice.paid`, `subscription.updated/deleted`) → синхронизация статуса тарифа в БД.
- Billing portal (смена карты/тарифа/отмена) — ссылка из личного кабинета.

### 1.5 Документация API
- OpenAPI/Swagger (springdoc) + страница «API» в `web` с примерами (curl, JS, GitHub Action).

**Критерий готовности:** внешний клиент с API-ключом сканирует через REST; вызовы считаются;
превышение квоты → 429; подписка/овередж списываются через Stripe; есть публичные API-доки.

## Phase 2 — Pro-тир (подписка для разработчиков и админов серверов)

> Платят за то, что бесплатному пользователю не нужно, а профи нужно постоянно.

### 2.1 Личный кабинет и история
- Next.js dashboard: список своих сканов (Phase 0 БД), фильтры, повторное открытие отчёта,
  сравнение версий одного плагина (diff findings между сканами).

### 2.2 Динамическая песочница как платная фича
- Сейчас `pluginguard.sandbox.enabled` глобальный. Сделать **per-request / per-plan**:
  бесплатный → только статика; Pro/Business → можно запросить sandbox-проход.
- Гейтить в `ScanController`/`SandboxService` по тарифу из контекста запроса.
- Это естественный paywall: дорогой Docker-compute = платный.

### 2.3 Bulk-скан
- Загрузка нескольких файлов / архива папки `plugins/` → пакет job'ов → сводный отчёт.
- Эндпоинт `POST /api/scan/batch`; в UI — drag папки.

### 2.4 «Verified»-бейдж для разработчиков
- Разработчик сканирует свой релиз → получает публичный отчёт + встраиваемый бейдж
  (SVG/markdown «Scanned by PluginGuard ✓»). Тщеславие + доверие + дистрибуция бренда.

**Критерий готовности:** Pro-пользователь видит историю, гоняет песочницу и bulk-скан, может
поставить бейдж; бесплатный — только синхронная статика одного файла.

## Phase 3 — Continuous monitoring и threat-intelligence (повторяющаяся ценность)

> Самая «липкая» (retention) часть: ценность приходит со временем, без действий пользователя.

### 3.1 Реестр «установленного»
- Пользователь регистрирует набор плагинов (загрузка или хеши/имена) как «watchlist» сервера.

### 3.2 Re-scan по новому threat-intel
- Фоновая джоба: при обновлении CVE (OSV.dev — уже есть `CveAnalyzer`) или сигнатур
  (`MalwareSignatureAnalyzer`, `known-malicious`-фид) — перепроверяет watchlist по сохранённому
  артефакту (Phase 0.2) **без повторной загрузки пользователем**.
- Кейс-убийца: «Плагин X v1.2 на твоём сервере — для него только что вышел CVE / он попал в
  список вредоносных».

### 3.3 Алерты
- Каналы: email, **webhook**, **Discord** (ниша живёт в Discord). Шаблон уведомления + дедуп.

### 3.4 Threat-intel фид как продукт
- `known-malicious`/`known-good` хеши + family-сигнатуры → подписной фид (по модели антивирусных
  баз). Покупатели: маркетплейсы, хостинги, другие инструменты.
- Отдельный эндпоинт `GET /api/intel/...` под отдельным scope ключа.

**Критерий готовности:** watchlist перепроверяется по расписанию/при апдейте фида; алерт
доходит до канала; threat-фид отдаётся по подписке отдельным ключом.

## Phase 4 — Дистрибуция и интеграции (рост и верхушка воронки)

### 4.1 GitHub Action (приоритет — дёшево + вирусно)
- Action `pluginguard/scan-action`: на build/PR сканирует артефакт через API, печатает
  findings, **падает по порогу score** (gate). Бейдж в README репозитория = бесплатная реклама.

### 4.2 Интеграции для маркетплейсов/хостингов
- **scan-on-upload** (маркетплейс): вебхук при заливке плагина → авто-скан → вердикт в карточку.
- **scan-on-install** (хостинг): хук панели при установке плагина клиентом.
- White-label виджет «Проверено PluginGuard» для встраивания в чужую страницу.

### 4.3 Прямой аутрич (не код, но часть плана)
- Цель в первую очередь: **1–2 маркетплейса/хостинга**. Один B2B-контракт > сотни Pro-подписок.
- Кандидаты: маркетплейсы (BuiltByBit/Polymart/Modrinth/SpigotMC), хостинги
  (Apex/Shockbyte/BisectHosting/PebbleHost).

**Критерий готовности:** GitHub Action в маркетплейсе Actions; есть рабочая webhook-интеграция и
хотя бы один пилот с внешней площадкой.

## Phase 5 — Enterprise / white-label

### 5.1 Self-hosted лицензия
- Лицензионный ключ, оффлайн-режим (есть `offline`), без телеметрии. Air-gapped инсталляция
  продаётся хостингам и крупным сетям (privacy proprietary-плагинов — сильный аргумент).

### 5.2 Корпоративный контур
- SSO (SAML/OIDC), audit log, роли/команды, кастомные правила и приватные threat-фиды, SLA.

### 5.3 White-label
- Полный ребренд отчёта/виджета под площадку-партнёра.

**Критерий готовности:** клиент разворачивает PluginGuard в своём контуре по лицензии, оффлайн,
со своими правилами и брендингом.

## Карта тарифов (черновик, цифры иллюстративные)

| Тир | Кому | Что входит | Цена (ориентир) |
|-----|------|------------|------------------|
| **Free** | всем | синхронная статика 1 файла, публичный отчёт, бейдж | $0 навсегда |
| **Pro** | разработчики, админы | песочница, история, bulk, мониторинг watchlist | ~$5–15/мес |
| **Business / API** | хостинги, маркетплейсы, CI | API с квотой + овередж, приоритет, webhook | usage-based |

| **Enterprise** | крупные / on-prem | self-host, SSO, audit, кастом-правила, SLA, white-label | контракт |

## Порядок работ (по ROI)

1. **Phase 0** — фундамент (БД, хранилище, jobs, разнесённый деплой). Без него ничего не работает.
2. **Phase 1** — аккаунты + API-ключи + учёт + Stripe. Открывает B2B = деньги.
3. **Phase 4.1** — GitHub Action (параллельно с Phase 1: дёшево, максимум дистрибуции).
4. **Phase 2** — Pro-фичи (песочница за paywall, история, bulk, бейдж).
5. **Phase 3** — мониторинг + threat-фид (retention и второй денежный поток).
6. **Phase 4.2–4.3** — интеграции и аутрич к площадкам.
7. **Phase 5** — enterprise/white-label под конкретного крупного клиента.

> Параллелизм: Phase 0 — обязательный пролог. Phase 1 и 4.1 идут вместе. Phase 5 — реактивно,
> «под спрос», не строить заранее.

## Риски и оговорки

- **Абуз бесплатного тира**: лимит размера файла + rate-limit по IP/ключу, иначе бесплатным
  сканом гоняют чужой CI и жгут твой compute. Песочница — строго за paywall.
- **Стоимость песочницы**: один вирусный пост может обнулить бюджет, если sandbox бесплатна.
- **False positives** в бесплатном тире = удар по доверию (там качество = маркетинг).
- **Юридика**: ToS + «risk assessment, не гарантия»; политика хранения загруженных артефактов
  (приватность proprietary-кода).
- **Доверие к open-source**: открытость движка — актив, а не угроза; ров — данные + хостинг.

## Definition of Done (для всего трека)

- Любой может бесплатно проверить один плагин (статика) без аккаунта — навсегда.
- Есть платный API с ключами, квотами и оплатой по факту.
- Pro даёт песочницу, историю, bulk, мониторинг.
- Threat-фид и self-host продаются отдельно.
- Каждая платная фича лежит **выше** линии «базовая безопасность бесплатна».
